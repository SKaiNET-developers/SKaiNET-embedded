= How to self-compile the Moonshine encoder
:description: Replace the prebuilt vendor encoder.vmfb with one the demo builds from the NN DSL.

The ASR path drives three vmfbs: a preprocessor (CPU), the **encoder** (NPU), and the
decoder pair (NPU). The encoder is authored in the SKaiNET NN DSL
(`skainet-transformers-inference-moonshine`), so the demo can build it itself instead of
shipping Synaptics' opaque `encoder.vmfb`. (The decoder is not yet in the DSL ‚Äî it stays
prebuilt for now.)

== 1. Emit the encoder StableHLO from the DSL (host)

[source,bash]
----
# Runnable, board-shaped encoder: real weights baked in + board [1,dim,frames] layout:
CHECKPOINT=/path/to/moonshine-tiny-weights ENC_FRAMES=207 ENC_BOARD_LAYOUT=1 \
  ./gradlew moonshineEncoderMlir          # ‚Üí build/mlir/moonshine-encoder.mlir

# structure-only bring-up variants:
./gradlew moonshineEncoderMlir                          # 6 layers, bf16, placeholder weights
ENC_LAYERS=1 ENC_DTYPE=FP32 ./gradlew moonshineEncoderMlir
----

This authors `moonshineEncoder()`, traces it, runs the core dtype pass, and emits portable
StableHLO ‚Äî no target-specific ops (the Torq specifics are compile flags, below).

* `CHECKPOINT=<dir>` bakes real weights (per-tensor little-endian f32 `.bin`, named by the HF
  tensor name) as graph constants ‚Üí a runnable vmfb. See `MoonshineWeights.kt` for the HF-name
  mapping and the layout conventions (projections `arg = onnx·µÄ`, `fc1`/`fc2` transposed).
* `ENC_BOARD_LAYOUT=1` + `ENC_FRAMES=207` make the entry accept the board's `[1, dim, 207]`
  preprocessor output and transpose it to `[1, frames, dim]` inside the graph ‚Äî a drop-in for
  `MoonshineDecoder`'s encoder call.

== 2. Compile to a Torq vmfb ‚Äî in Docker (no host toolchain)

The IREE/Torq compiler runs in an image, so the only host requirement is Docker. Build the
image once (needs the private `torq_compiler` wheel ‚Äî not on PyPI):

[source,bash]
----
cp /path/to/torq_compiler-*.whl scripts/.docker/
docker build -f scripts/.docker/Dockerfile.iree -t sl2610-iree:local scripts/.docker
----

Then compile:

[source,bash]
----
scripts/iree-compile-torq-docker.sh build/mlir/moonshine-encoder.mlir build/mlir/encoder.vmfb
----

The image bakes the required `--torq-disable-slices` flag and the `ld ‚Üí iree-lld` shim
(a Torq vmfb links aarch64 host code the system linker rejects).

== 3. Deploy and select it on the board

[source,bash]
----
scp build/mlir/encoder.vmfb root@BOARD:/home/root/moon/encoder-selfcompiled.vmfb
# run the board binary with the env override:
MOONSHINE_ENCODER_VMFB=/home/root/moon/encoder-selfcompiled.vmfb voicecc pipeline test.wav
----

`MoonshineDecoder` picks up `MOONSHINE_ENCODER_VMFB` and uses our encoder in place of the
vendor one; everything else (preprocessor, decoder) is unchanged.

== Validation caveats

[IMPORTANT]
====
Weight-baking and the board-layout transpose are *implemented* (`CHECKPOINT` / `ENC_BOARD_LAYOUT`),
but two things still need attention:

. **Requires transformers 0.34.1.** The baker maps by NAME using the layer-qualified parameter
  names introduced in 0.34.1 (`enc.$layer.attn.q_proj.weight`, ‚Ä¶). The demo pins 0.34.0 until
  0.34.1 lands on Maven Central ‚Äî then bump `skainetTransformers` to `0.34.1`. Run against 0.34.0's
  un-qualified names, the baker fails fast with a clear message.
. **HF tensor names.** The checkpoint‚ÜíDSL mapping in `MoonshineWeights.kt` (`hfNameFor`) uses the
  expected moonshine-tiny HF names; verify them against your actual checkpoint (values are
  untestable without it). Missing tensors fail fast.
====
